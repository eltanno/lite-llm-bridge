# Quick Task 260522-i4q

## Objective

Prove, in isolation (this project only — no global shell/`~/.bashrc` changes), that a single
running LiteLLM gateway can expose **both** an Anthropic-backed alias (via Claude Max OAuth-token
forwarding) and Ollama-Cloud-backed aliases at the same time, and that Claude Code can drive both.

Eventual goal (OUT OF SCOPE here): promote this to the global default for all folders when running `claude`.

## Test model mapping (user choice, 2026-05-22)

| Alias  | Backend                          | Provider string                      | Auth                         |
|--------|----------------------------------|--------------------------------------|------------------------------|
| opus   | DeepSeek V4 Pro (Ollama Cloud)   | `openai/deepseek-v4-pro`             | `OLLAMA_API_KEY`             |
| sonnet | Kimi K2.6 (Ollama Cloud)         | `openai/kimi-k2.6`                   | `OLLAMA_API_KEY`             |
| haiku  | Claude Haiku 4.5 (Anthropic Max) | `anthropic/claude-haiku-4-5-20251001`| Max OAuth bearer (forwarded) |

OAuth forwarding scoped to `haiku` only via `model_group_settings.forward_client_headers_to_llm_api`.

## Tasks

1. Rewrite `config.yaml` with the mapping above + scoped header forwarding. Atomic commit.
2. Fix the stale `CLAUDE.md` audit-table line ("Claude Max routes through LiteLLM | REFUTED") to
   match verified reality. Atomic commit.
3. `docker compose restart` so LiteLLM reloads config; confirm it comes up and exposes opus/sonnet/haiku.
4. Verify both backends non-interactively with stored Max creds:
   - `claude -p --model haiku ...` → routes through Anthropic/Max (confirm via gateway logs).
   - `claude -p --model opus ...`  → routes through Ollama Cloud (confirm via gateway logs).
   Gateway env (`ANTHROPIC_BASE_URL`, `ANTHROPIC_CUSTOM_HEADERS`=master key, `ANTHROPIC_DEFAULT_*_MODEL`;
   `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` unset) set ONLY in the test subprocess.

## must_haves

- truths:
  - A single gateway config serves an Anthropic-Max alias and an Ollama alias simultaneously.
  - `claude -p --model haiku` returns a successful completion routed to Anthropic (Max), not Ollama.
  - `claude -p --model opus` returns a successful completion routed to Ollama, not Anthropic.
  - The Ollama aliases never receive the Max OAuth bearer (scoping holds, or E-01 fallback applied safely).
- artifacts:
  - `config.yaml` with the three aliases + `model_group_settings` scoping.
  - Gateway-log evidence of correct per-alias routing.

## Risk / E-01 gate

`model_group_settings.forward_client_headers_to_llm_api: [haiku]` is the documented per-group syntax,
but was never empirically confirmed on the pinned `main-stable` build. If `opus` (Ollama) fails with a
401 after restart (OAuth bearer leaking to Ollama), fall back to `general_settings:
forward_client_headers_to_llm_api: true` — safe because no `ANTHROPIC_API_KEY` is in the container.

## Isolation guarantees

- No edits to `~/.bashrc` or global `~/.claude`.
- Confirmed this Claude Code session is NOT gateway-routed, so `docker compose restart` won't disrupt it.
- Test env vars live only in the verification subprocess.
